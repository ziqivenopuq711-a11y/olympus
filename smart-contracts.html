Protocol Architecture (Default Framework)

Kernel, Modules & Policies

Olympus V3 uses the Default Framework — a modular architecture that separates protocol logic into three layers. Understanding this architecture is essential for finding and reporting vulnerabilities correctly.

KERNEL
Kernel.sol
The central registry that manages module and policy permissions. Unauthorized kernel access or role escalation are critical-severity findings.
MODULE
TRSRY — Treasury
Manages protocol-owned reserves and withdrawals
MODULE
MINTR — Minter
Controls OHM minting and burning authorization
MODULE
PRICE — Price Oracle
Aggregates and validates on-chain price feeds
MODULE
RANGE — RBS Range
Manages cushion and wall price levels for stability
MODULE
ROLES — Role Manager
Stores and validates access roles for policies
MODULE
VOTES — Voting Power
Tracks OHM-based governance voting weight
POLICY
Policy Contracts (BondCallback, Governance, Operator, etc.)
Policies implement protocol logic by reading from and writing to modules via the Kernel. Improper policy-to-module interactions are a common vulnerability class.

Accepted Impact Categories

Critical
Unauthorized extraction of treasury assets
Unauthorized OHM or gOHM minting
Direct theft of staked user funds
Kernel role escalation or bypass
High
Bond pricing manipulation (discounted acquisition)
Oracle price feed manipulation affecting RBS
Unauthorized governance proposal execution
Permanent freezing of staking rewards

Out-of-Scope Contracts & Impacts

The following contracts and vulnerability categories are explicitly out of scope. Submissions relating to out-of-scope items will not receive rewards:

  • Any third-party DeFi protocol integrations (Uniswap, Balancer, Aave, Curve) not deployed or maintained by OlympusDAO
  • Findings in legacy V1/V2 contracts that are no longer in production use
  • Yield optimization strategies that result in internal rebalancing without value extraction
  • Governance manipulation requiring a majority of gOHM supply (>50% governance attack)
  • Gas optimization suggestions, code quality improvements, or best-practice deviations without security impact
  • Bugs in previously completed audit reports that have not been fixed (these are known issues)
  • Issues in deployed contracts on testnets, as they are not the canonical production versions

How to Report a Smart Contract Vulnerability

To report a smart contract vulnerability in the Olympus protocol, follow these steps to ensure your submission is considered and processed efficiently:

  • Reproduce the bug on a local fork — use Foundry, Hardhat, or Tenderly fork to create a reproducible PoC.
  • Identify the exact contract and function — specify the contract name, address, and the function(s) involved.
  • Quantify the impact — estimate the maximum amount of funds at risk in USD terms.
  • Write a clear exploit scenario — describe step-by-step how an attacker would exploit the bug in practice.
  • Attach your PoC code — include the full test file that demonstrates the exploit.
  • Submit via Immunefi — all submissions must go through the official Immunefi portal linked below.
