Overview / Testnet

Testnet Environment

Guidelines for setting up and using a local fork environment to safely test and reproduce vulnerabilities against Olympus protocol contracts.

Testing on Live Networks is Prohibited

Any testing on mainnet or public testnets is strictly forbidden. All research must be conducted on a local fork of mainnet. Submissions that include evidence of live-network testing will be disqualified.

Setting Up a Local Fork

Recommended Setup with Foundry / Anvil
Use a forked local environment that mirrors mainnet state. Foundry's Anvil and Hardhat Network both support mainnet forking via a JSON-RPC provider such as Alchemy or Infura.
1
Install Foundry 
curl -L https://foundry.paradigm.xyz | bash
foundryup
2
Fork Ethereum Mainnet with Anvil 
anvil --fork-url https://eth-mainnet.g.alchemy.com/v2/YOUR_KEY \
      --fork-block-number 19000000
3
Clone OlympusDAO contracts repository 
git clone https://github.com/OlympusDAO/olympus-contracts
cd olympus-contracts
forge install
4
Run your PoC test against the fork 
forge test --fork-url http://127.0.0.1:8545 \
           --match-test testMyExploit -vvv
5
Document and submit your findings
Include test output, transaction traces, and a clear description of the impact. Submit via the Immunefi portal.

Testing Rules & Best Practices

Security researchers must adhere to the following testing guidelines. Violations may result in permanent disqualification from the program and potential legal action.

  • Use local forks only — fork mainnet state using Anvil, Hardhat, or Tenderly. Never submit transactions to mainnet or public testnets.
  • Do not test against live oracle price feeds — use mock prices or snapshot values in your local environment.
  • Do not generate excessive RPC traffic — automated scanning tools that hammer public endpoints are prohibited.
  • Isolate your testing environment — ensure your PoC does not interact with any third-party protocols not maintained by OlympusDAO.
  • Document your environment — include the block number forked, Foundry/Hardhat version, and any relevant environment variables in your submission.
  • Responsible disclosure — do not discuss findings publicly before receiving written approval from the Bug Bounty Management team (Category 3: Approval Required).

The Olympus team is available on Discord for clarifying questions about scope or environment setup. Do not share vulnerability details in public channels — use the private #bug-bounty channel.

PoC Submission Checklist

Before submitting your finding, ensure your PoC includes all of the following:

Working Foundry / Hardhat test file
Fork block number & network specified
Clear description of the vulnerability
Estimated financial impact (USD)
Test output / transaction trace included
Affected contract(s) identified
